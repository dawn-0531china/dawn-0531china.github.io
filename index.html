WKWebView的艰难之路

1、Cookie的设置

1.1、WKWebView loadRequest前，在request header中设置Cookie，解决首个请求Cookie带不上的问题
1.2、通过系统API设置cookie

很多技术文章说WKWebView的Cookie数据和HTTPCookieStorage是分离的。通过实测WKWebView可以读到HTTPCookieStorage中的Cookie中的数据，但在时机上会存在延时。

1.3、通过注入本地JS脚本设置Cookie

通过JS脚本设置cookie，需要在configuration赋值给webView前就将脚本注入

1.4、重定向下的Cookie设置

当WKWebView发生跨域302重定向请求时，Cookie时无法同步到新的请求链接上去的。当然，由于每一次页面跳转都会调用回调函数：

可以在该回调里拦截302请求，copy request，在request header中带上cookie并重新loadRequest。不过这种方法依然无法解决页面iframe跨域请求的Cookie问题，毕竟[WKWebView loadRequest:]只适合mainFrame请求。

2、UA

2.1、全局UA的设置

WKWebView全局UA的生成是通过在AppDelegate创建一个WKWebView的实例，通过evaluateJavaScript(“navigator.userAgent”)读取到已有的UA，进行处理后（加入应用标示、版本等信息），保存到NSUserDefaults中。
在设置全局UA后，再通过evaluateJavaScript(“navigator.userAgent”)读取到的UA信息已经是设置的全局UA，因此在处理的时候要注意下。

在AppDelegate中应用启动后进行设置

全局UA的设置过程是一个异步过程，在应用启动后立刻打开网页会存在UA设置不成功的情况，在开发的时候一定要注意。

2.2、通过系统API对不同的WKWebView设置UA

WKWebView自有的设置UA的API在iOS 12.0 ～ iOS 12.2的系统中是一个异步的设置过程，直接使用可能会造成在这些系统范围的设备UA设置不成功的情况。

在单独设置WKWebView的UA时，先判断全局的UA是否设置成功，如果成功就读取全局UA对customUserAgent进行赋值，如果全局的UA没有设置成功，可以通过evaluateJavaScript(“navigator.userAgent”)进行单独的UA设置。

3、缓存

3.1、URLProtocol 设置缓存

通过注册https、http协议，可以在遵从URLProtocol协议的实例中拦截到所有的网络请求，在请求响应成功后对资源数据进行缓存处理。在处理静态网页资源缓存上这样处理是没有问题。
但是在存在POST网络请求的网页使用URLProtocol进行资源缓存就会造成POST请求的body数据丢失的情况。
由于WKWebView是一个单独的线程进行处理网络请求的，在使用URLProtocol注册https、http后，就会将请求转发到应用的进程中，处于性能考虑，会将POST请求的body抛弃。
在使用URLProtocol时，不要注册https、http。

3.2、URLRequest.CachePolicy

初始化URLRequest的时候可以指定缓存策略使用协议缓存useProtocolCachePolicy。
在服务端返回资源时，会将资源的最后更新资源通过Last-Modified返回给客户端，客户端再次发起请求时通过If-Modified-Since或者If-Unmodified-Since带上Last-Modified，服务端检查该时间是否和服务端资源的最后更新时间是否一致：如果一致，则返回304状态码，不返回资源；如果不一致，则返回200状态码和修改后的资源，并带上资源最后的修改时间。

4、WKWebView内自动播放音视频

如果WKWebView和原生的交互使用prompt弹窗同步交互方法，会存在打断音视频自动播放的情况。如果有音视频自动播放的需求，在交互方法上可以选用系统提供的WKScriptMessageHandler进行交互。

Grand Central Dispatch

发表于 2020-06-02 | 分类于 iOS

0、前言

三年前某君共享了几本电子书给我，记得有一本里面介绍了GCD。近来想重新温习一下，电子书却已经找不到了，就又找某君让他共享了下。
温故知新，现在再看这本书中的GCD确实有新的收获。第一次看的感受、是否有获取到里面的点已经记得不清了，也不能说自己完全掌握里面的知识点，但是重新温习已经看过文章或者书籍确实能有全新的感觉。
原本这篇笔记是记录在Evernote中的，却被自己拿来调试GitHub个人博客，成为了自己个人博客的第一篇文章，就写个前言记录下吧。

1、Grand Central Dispatch （GCD）概要

1.1、什么是GCD

是异步执行任务的技术之一。开发者只需要定义想要执行的任务并追加到适当的GCD中，就能生成必要的线程并计划执行任务。由于线程管理是作为系统的一部分实现的，因此可以统一管理，也可执行任务。

1.2、多线程编程
  1. CPU执行一个的CPU命令列是一条无分叉的路径，即为线程
  2. CPU执行多条不同路径上的不同命令，即为多线程
  3. 1个CPU核一次能够执行的CPU命令始终为1
  4. CPU在执行多线程程序时，能在某个线程和其他线程之间反复多次的上下文切换，因此看上去好像1个CPU能够并列地执行多个线程一样
  5. 多线程是一种易发生各种问题的编程技术。例如：
    • 多个线程更新相同的资源会导致数据的不一致（数据竞争）
    • 停止等待事件的线程会导致多个线程相互持续等待（死锁）
    • 使用太多线程会消耗大量内存
  6. 主线程用来描绘用户界面、处理触摸屏幕的事件，在主线程中进行长时间的处理，会妨碍主线程的执行（阻塞）
  7. 使用多线程执行长时间的处理，可以保证用户界面的响应性能

2、GCD的API

2.1、Dispatch Queue
  1. Dispatch Queue 指的是执行处理的等待队列。将想要执行的处理追加到Dispath Queue中，Dispatch Queue会按照追加的顺序（先进先出FIFO，First-In-First-Out）执行处理
  2. 执行处理时存在两种Dispatch Queue：一种是等待现在执行中的处理 Serial Dispatch；另一种是不等待现在执行中的处理 Concurrent Dispatch Queue
Dispatch Queue 的种类 说明
Serial Dispatch Queue 等待现在执行中的任务处理结束
Concurrent Dispatch Queue 不等待现在执行中的任务处理结束
  1. 当Dispatch Queue为Serial Dispatch Queue时要等待当前任务执行结束后才能执行下一个任务，所以使用的是一个线程执行任务
  2. 当Dispatch Queue为Concurrent Dispatch Queue时不用等待当前任务执行结束就可以执行下一个任务，所以使用的是多个线程执行任务
2.2、dispatch_queue_create
  1. GCD的API生成Dispatch Queue
    • 第一个参数指定Dispatch Queue的名称。推荐使用应用程序ID这种逆序全程域名
    • 第二个参数指定为NULL或者DISPATCH_QUEUE_SERIAL时生成Serial Dispatch Queue；为DISPATCH_QUEUE_CONCURRNET时生成Concurrent Dispatch Queue
  2. 生成Serial Dispatch Queue 时的注意事项
    • dispatch_queue_create 可以生成多个Serial Dispatch Queue，各个Dispatch Queue将并行执行
    • 一个Serial Dispatch Queue 一次只能追加一个处理，一但追加处理系统将会生成并使用一个线程。如果生成2000个Serial Dispatch Queue，那么就生成2000个线程
    • 过多的使用线程，就会消耗大量内存，引起大量的上下文切换，大幅降低系统的响应性能。绝对不能大量的生成Serial Dispatch Queue
    • 只在避免多线程更显相同资源导致数据竞争时使用Serial Dispatch Queue
    • 不考虑数据竞争问题时，对于Concurrent Dispatch Queue 来说，不管生成多少，由于XNU内核只使用有效管理的线程，因此不会出现Serial Dispatch Queue的那些问题
2.3、Main Dispathc Queue / Global Dispatch Queue
  1. Main Dispatch Queue 是在主线程中执行的Dispatch Queue。主线程只有一个，所以Main Dispatch Queue是Serial Dispatch Queue
  2. Global Dispatch Queue 是所有应用程序都能够使用的Concurrent Dispatch Queue
  3. Global Dispatch Queue 有四个执行优先级，分别是高优先级（High Priority）、默认优先级（Default Priority）、低优先级（Low Priority）、后台优先级（Background Priority）
名称 Dispatch Queue 的种类 说明
Main Dispatch Queue Serial Disaptch Queue 主线程执行
Global Dispatch Queue（High Priority） Concurrent Dispatch Queue 执行优先级：高（最高优先级）
Global Dispatch Queue（Default Priority） Concurrent Dispatch Queue 执行优先级：默认
Global Dispatch Queue（Low Priority） Concurrent Dispatch Queue 执行优先级：低
Global Dispatch Queue（Background Priority） Concurrent Dispatch Queue 执行优先级：后台

2.5 Dispatch Group
  1. 在追加到Dispatch Queue中的多个处理全部结束后想执行结束处理，这个时候可以使用Dispatch Group
  2. 无论向什么样的Dispatch Queue中追加处理，使用Dispatch Group都可以监视这些处理执行结束。一旦检测到所有的处理执行结束，就可以将结束处理追加到Dispatch Queue中

  • 如果返回值不为0，意味着虽然经过了指定时间，但属于Dispatch Queue的某一个处理还在执行中；如果返回值为0，那么全部处理执行结束
2.6、dispatch_barrier_async/dispatch_barrier_sync
  1. dispatch_barrier_async函数会等待追加到Concurrent Dispatch Queue上的并行处理执行全部结束后，再将指定的处理追加到该Concurrent Dispatch Queue中。然后由dispatch_barrier_async函数追加的处理执行完毕后，Concurrent Dispatch Queue才恢复为一般的动作
  • 如上图所示，使用dispatch_barrier_async追加到Concurrent Dispatch Queue上的处理能有效的解决多线程引起的数据竞争问题
  1. dispatch_barrier_sync函数，提交一个栏栅函数在Dispatch Queue中，它会等待栏栅函数执行完，才会执行栏栅函数后面的任务
  2. dispatch_barrier_sync函数是在当前线程中执行追加到Dispatch Queue中的任务，而dispatch_barrier_async函数时开辟了分线程执行追加到Dispatch Queue中的任务
  3. Dispatch Barrier API 提供了高效的读写方案；只有将Barrier Blocks提交到自己创建的Concurrent Dispatch Queue才会有特殊的行为；在这样的队列上，直到前面提交到队列的任务都完成后，barrier blocks才会执行，而在barrier blocks之后提交到队列中的任务，在barrier blocks完成之前都不会运行
  4. 当提交到全局队列或者没有使用DISPATCH_QUEUE_CONCURRENT创建的队列时，barrier blocks的行为与使用dispatch_async/dispatch_sync提交的blocks相同
2.7 dispatch_sync / dispatch_async
  1. dispatch_async函数“async”意味着“非同步”，就是将指定Block“非同步”地追加到指定的Dispatch Queue中，dispatch_async函数不做任何等待
  2. dispatch_sync函数“sync”意味着“同步”，就是将指定Block“同步”地追加到指定的Dispatch Queue中，在追加Block结束之前，dispatch_sync函数会一直等待
  3. 一旦调用dispatch_sync函数，那么在指定的处理执行结束之前，该函数不会返回

2.8、dispatch_apply
  1. dispatch_apply函数时dispatch_async函数和Dispatch Group的关联API。该函数按指定的次数将指定的Block追加到指定的Dispatch Queue中，并等待全部处理执行结束
  • 第一个参数为重复的次数，第二个参数为追加对象的Dispatch Queue，第三个参数为追加的处理
  1. 由于dispatch_apply函数与dispatch_sync函数相同，会等待处理执行结束，因此推荐在dispatch_async函数中非同步地执行dispatch_apply函数
2.9、dispatch_suspend / dispatch_resume
  1. dispatch_suspend函数挂起指定的Dispatch Queue
  2. dispatch_resume函数恢复指定的Dispatch Queue
  3. 这些函数对已经执行的处理没有影响。挂起后，追加到Dispatch Queue中但尚未执行的处理在此后停止执行。而恢复则使得这些处理能够继续执行
2.10、Dispatch Semaphore
  1. Dispatch Semaphore 时持有计数的信号，该计数时多线程编程中的计数类型信号。计数为0时等待，计数为1或大于1时，减去1而不等待
  1. dispatch_semaphore_wait函数等待Dispatch Semahpore的计数值达到大于或者等于1。当计数值大于等于1，或者在待机中计数值大于等于1时，对该计数进行减去并从dispatch_semaphore_wait函数返回。返回值为0时，可以安全地执行需要进行排它控制的处理。处理结束时通过dispatch_semaphore_signal函数将Dispatch Semaphore的计数值加1。
2.11、dispatch_once

dispatch_once函数时保证在应用程序执行中只执行一次指定处理的API。通过dispatch_once函数，即使在多线程环境下执行，也可以保证百分之百的安全。

4、总结

最主要的内容是介绍了GCD API的使用，而贯穿整个介绍API使用过程的是同步和异步的思想。同步：会等待任务执行完成后才会进行后续代码的调用；异步：不需要等待任务执行完成就可以进行后续代码的调用。因此在使用的过程不仅仅要考虑是否开启线程、是否在主线程中执行，还要考虑调用的API是同步执行任务还是异步执行任务，这对GCD的使用是有很大益处的。
